# /learned - 提取、评估、然后保存

分析当前会话，经过质量门控、保存位置决策和知识放置意识。提取值得保存为技能的任何模式。

## 提取内容

寻找：

1. **错误解决模式** — 根本原因 + 修复方法 + 可重用性
2. **调试技术** — 非显而易见的步骤、工具组合
3. **变通方法** — 库的怪癖、API 限制、特定版本的修复
4. **项目特定模式** — 约定、架构决策、集成模式

## 流程

### 第一步：识别与回顾

1. 识别用户使用 /learned 命令后面提到的补充想法
2. 回顾会话，寻找可提取的模式，识别最有价值/可重用的见解

### 第二步：质量门控

#### 必需清单（通过实际阅读文件进行验证）

在评估草案**之前**，执行以下所有操作：

- \[ ] 使用关键字在 `~/.claude/skills/` 和相关项目的 `.claude/skills/` 文件中进行 grep 搜索，检查内容重叠【注：可能是codex】
- \[ ] 检查 MEMORY.md（项目级和全局级）以查找重叠内容
- \[ ] 考虑是否追加到现有技能即可满足需求
- \[ ] 确认这是一个可复用的模式，而非一次性修复

#### 整体裁决

综合清单结果和草案质量，然后选择**以下一项**：

| 裁决 | 含义 | 下一步行动 |
|------|------|------------|
| **保存** | 独特、具体、范围明确 | 进行到第四步 |
| **改进后保存** | 有价值但需要改进 | 列出改进项 → 修订 → 重新评估（一次） |
| **吸收到 \[X]** | 应追加到现有技能 | 显示目标技能和添加内容 → 第四步 |
| **放弃** | 琐碎、冗余或过于抽象 | 解释原因并停止 |

**指导维度**（用于告知裁决，不进行评分）：

- **具体性和可操作性**：包含可立即使用的代码示例或命令
- **范围契合度**：名称、触发条件和内容保持一致，并专注于单一模式
- **独特性**：提供现有技能未涵盖的价值（基于清单结果）
- **可复用性**：在未来的会话中存在现实的触发场景

### 第三步：裁决确认

给出简短初步判断，询问用户侧重点，以及建议。根据用户反馈调整，或者不需要调整，进行下一步。

- **保存**：呈现保存路径 + 清单结果 + 1 行裁决理由 + 完整草案 → 在用户确认后保存
- **改进后保存**：呈现改进项 + 修订草案 + 重新评估；如果修订后裁决为**保存**，则在用户确认后保存，否则遵循新裁决
- **吸收到 \[X]**：呈现目标路径 + 添加内容（diff 格式） + 清单结果 + 裁决理由 → 在用户确认后追加
- **放弃**：仅显示清单结果 + 推理（无需确认）

### 第四步：根据模板起草技能文件

起草或追加技能内容前，必须先读取模板文件，依照模板文件的格式书写：

- 模板路径：`references/learned-skill-template.md`

格式要求：

- 变更记录中的“原因 / 背景”必须写在表格中，不要额外新增独立原因段落。
- “整体架构的链路”优先按模板中的多节点骨架书写，但节点名称和数量要按实际功能调整；关键节点要标注输入、状态/数据变化、判断、动作和结果确认。
- “调试入口”必须包含两个小节：完整 LOG 正则表达式、关键 LOG 短句释义。
- 完整 LOG 正则表达式只放可复制使用的正则文本；关键 LOG 短句释义使用表格说明每条日志代表的状态、消息含义或成功/失败信号。

### 第五步：保存

- **默认保存到用户目录**（`~/.claude/skills/learned-*/SKILL.md`），除非用户明确要求保存到项目目录
- **项目目录**（`.claude/skills/learned-*/SKILL.md`）：仅当用户明确指定时使用
- 即使是项目特定的知识，也默认放在用户目录，用户有需要时可自行要求迁移

## 输出格式

流程结束后，输出以下摘要：

```
### 检查清单
- [x] skills/ grep: 无重叠 (或: 发现重叠 → 详情)
- [x] MEMORY.md: 无重叠 (或: 发现重叠 → 详情)
- [x] 现有技能追加: 新文件合适 (或: 应追加到 [X])
- [x] 可复用性: 已确认 (或: 一次性 → 丢弃)

### 裁决: 保存 / 改进后保存 / 吸收到 [X] / 丢弃

### 理由： (用 1-2 句话解释裁决)

### 简单描述本次沉淀的内容：

### 文件位置：
```

## 注意事项

- 不要提取琐碎的修复（拼写错误、简单的语法错误）
- 不要提取一次性问题（特定的 API 中断等）
- 专注于那些将在未来会话中节省时间的模式
- 保持技能聚焦 —— 每个技能一个模式
- 当裁决为"吸收"时，追加到现有技能，而不是创建新文件
