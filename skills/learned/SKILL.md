# Learned - 提取、评估、然后保存

当用户要求沉淀经验、保存 learned skill、提取当前会话模式、记录调试套路或类似知识复用请求时使用此技能。目标是从当前会话中提取值得保存的模式，并在保存前完成质量门控、重复检查和保存位置判断。

## 提取内容

寻找：

1. **错误解决模式** — 根本原因 + 修复方法 + 可重用性
2. **调试技术** — 非显而易见的步骤、工具组合
3. **变通方法** — 库的怪癖、API 限制、特定版本的修复
4. **项目特定模式** — 约定、架构决策、集成模式

## 流程

### 第一步：识别与回顾

1. 识别用户在本次请求中给出的补充想法、保存范围和命名偏好。
2. 回顾会话，寻找可提取的模式，优先选择最有价值、可重用、能在未来节省排查时间的见解。

### 第二步：质量门控

在评估草案**之前**，执行以下所有操作：

- [ ] 使用关键字在 `~/.claude/skills/`、`~/.codex/skills/` 和相关项目的 `.claude/skills/`、`.codex/skills/` 中进行搜索，检查内容重叠
- [ ] 检查 MEMORY.md（项目级和全局级，如果可用）以查找重叠内容
- [ ] 考虑是否追加到现有技能即可满足需求
- [ ] 确认这是一个可复用的模式，而非一次性修复

### 第三步：整体裁决

综合清单结果和草案质量，然后选择**以下一项**：

| 裁决 | 含义 | 下一步行动 |
|------|------|------------|
| **保存** | 独特、具体、范围明确 | 继续起草并请求确认 |
| **改进后保存** | 有价值但需要改进 | 列出改进项，修订后重新评估一次 |
| **吸收到 [X]** | 应追加到现有技能 | 展示目标技能和追加内容 |
| **放弃** | 琐碎、冗余或过于抽象 | 解释原因并停止 |

判断维度：

- **具体性和可操作性**：包含可立即使用的代码示例、命令、日志关键字或定位步骤
- **范围契合度**：名称、触发条件和内容保持一致，并专注于单一模式
- **独特性**：提供现有技能未涵盖的价值
- **可复用性**：未来会话中存在现实触发场景

### 第四步：裁决确认

根据裁决选择对应交互：

- **保存**：呈现保存路径 + 清单结果 + 1 行裁决理由 + 完整草案，在用户确认后保存
- **改进后保存**：呈现改进项 + 修订草案 + 重新评估；如果修订后裁决为**保存**，在用户确认后保存
- **吸收到 [X]**：呈现目标路径 + 添加内容（diff 格式）+ 清单结果 + 裁决理由，在用户确认后追加
- **放弃**：仅显示清单结果 + 推理，无需确认

### 第五步：起草技能文件

以下模板仅供参考，根据实际情况调整：

```markdown
---
name: pattern-name
description: "Under 130 chars"
origin: auto-extracted
---

# [描述性的技能名称]

**适用场景:** [简要描述何时使用此模式]

## 问题背景

[简单阐述事情经过]

## 入口

[触发起点：信号/消息/调用 + 文件:行号]

## 完整链路

[A -> B -> C -> D，每步标注文件、函数和关键参数]

## 关键判断点

[条件分支、配置项、状态检查 + 代码位置]

## 调试入口

[日志搜索关键字、grep 命令、断点位置]

## 常见问题排查

| 现象 | 原因 | 定位方法 |
|------|------|----------|
| [具体现象] | [根因] | [日志/命令/检查点] |
```

## 保存位置

- **默认保存到用户目录**：`~/.claude/skills/learned-*/SKILL.md` 或 `~/.codex/skills/learned-*/SKILL.md`，按当前工具环境选择。
- **项目目录**：`.claude/skills/learned-*/SKILL.md` 或 `.codex/skills/learned-*/SKILL.md`，仅当用户明确要求保存为项目知识时使用。
- 即使是项目特定知识，也默认放在用户目录，除非用户明确要求迁移到项目。

## 输出格式

流程结束后，输出以下摘要：

```markdown
### 检查清单
- [x] skills/ grep: 无重叠 (或: 发现重叠 -> 详情)
- [x] MEMORY.md: 无重叠 (或: 发现重叠 -> 详情)
- [x] 现有技能追加: 新文件合适 (或: 应追加到 [X])
- [x] 可复用性: 已确认 (或: 一次性 -> 放弃)

### 裁决: 保存 / 改进后保存 / 吸收到 [X] / 放弃

### 理由

### 简单描述本次沉淀的内容

### 文件位置
```

## 注意事项

- 不要提取琐碎的修复，例如拼写错误或简单语法错误。
- 不要提取一次性问题，例如临时 API 中断。
- 专注于未来会话中能节省时间的模式。
- 保持技能聚焦，每个技能只覆盖一个模式。
- 当裁决为“吸收”时，追加到现有技能，而不是创建新文件。
